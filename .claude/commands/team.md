# Team Agent Mode

사용자의 요청을 분석하고, 최적의 전문 팀을 자동 구성하여 협업으로 문제를 해결하는 팀 에이전트 모드입니다.

## ★★★ 3대 핵심 원칙 (절대 위반 금지) ★★★

### 원칙 1: 리더는 코딩하지 않는다 (작업 외주화)

나(리더, 메인 클로드)는 **지휘관**이다. 코드를 직접 읽지도, 쓰지도, 수정하지도 않는다.
- **리더가 하는 것**: 작업 분석, 팀원 배정, decisions.md 관리, 진행 상황 판단, 사용자 보고
- **리더가 절대 하지 않는 것**: 소스 코드 Read/Edit/Write, Grep으로 코드 검색, 직접 구현
- **모든 코드 작업**은 Task 도구로 팀원(서브에이전트)에게 위임한다
- 리더의 컨텍스트 윈도우는 전체 작전을 기억하는 유일한 공간이므로, 코드로 채우면 안 된다

### 원칙 2: 기억을 파일에 쓴다 (기억 외부화)

중요한 결정이 나올 때마다 즉시 `.claude/team-decisions.md`에 기록한다.
- 매 Phase 완료 시 decisions.md 업데이트 필수
- 팀원에게 작업 지시할 때 decisions.md의 내용을 컨텍스트로 전달
- 컨텍스트가 날아가도 이 파일만 읽으면 복구 가능
- 기록 항목: 작업 목표, 설계 결정, 완료된 작업, 변경된 파일, 발견된 이슈, 남은 작업

### 원칙 3: 팀원은 쓰고 버리고 새로 뽑는다 (계속 해고하기)

팀원(서브에이전트)은 일회용이다.
- 작업 끝난 팀원은 즉시 해고 (resume 하지 않음)
- 새 팀원에게는 decisions.md의 요약만 전달
- 하나의 팀원에게 너무 많은 작업을 주지 않는다 (1팀원 = 1작업)
- 팀원이 반환한 결과는 핵심만 추출하여 decisions.md에 기록

## Phase 0: 기억 저장소 초기화

**리더 액션**: `.claude/team-decisions.md` 파일을 생성/초기화한다.

Write 도구로 `.claude/team-decisions.md`에 다음을 기록:
```markdown
# Team Decisions Log
## 미션
- 원본 요청: [사용자 요청 원문]
- 시작 시각: [현재 시각]
## 작업 분석
(Phase 1에서 채움)
## 설계 결정
(Phase 3에서 채움)
## 완료된 작업
(진행하면서 채움)
## 변경된 파일
(진행하면서 채움)
## 발견된 이슈
(검토 시 채움)
## 남은 작업
(진행하면서 채움)
```

## Phase 1: 작업 분석 및 팀 구성

**리더만 수행** (팀원 배정 없음)

**작업 유형 자동 판단 기준**:

| 유형 | 키워드/패턴 | 구성 팀원 |
|------|------------|----------|
| **신규 개발** | 새로 만들어, 추가해, 구현해, 페이지 만들어 | 설계자 + 개발자 + 검토자 |
| **버그 수정** | 오류, 에러, 안됨, 수정해, 고쳐 | 분석가 + 개발자 + 검토자 |
| **리팩토링** | 개선, 리팩토링, 정리, 최적화 | 설계자 + 개발자 + 검토자 |
| **코드 리뷰** | 리뷰, 검토, 확인해줘 | 분석가 + 검토자 |
| **아키텍처** | 설계, 구조, 아키텍처 | 설계자 + 분석가 + 검토자 |
| **조사/분석** | 찾아줘, 분석해, 알려줘, 원인 | 분석가 + 검토자 |
| **DB 작업** | 테이블, 쿼리, DB, 데이터베이스 | DB전문가 + 개발자 + 검토자 |
| **UI/UX** | 디자인, 화면, 레이아웃, CSS | UI전문가 + 개발자 + 검토자 |

**리더 액션**:
1. 요청을 분석하여 작업 유형/복잡도 판단
2. 팀 구성 결정
3. 사용자에게 보고:
```
📋 작업 분석 결과
━━━━━━━━━━━━━━━━━━━━━━━━━━━
작업 유형: [판단된 유형]
복잡도: [단순/보통/복잡]
구성 팀원: [팀원 목록과 역할]
작업 계획: [단계별 실행 계획]
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
4. **decisions.md 업데이트**: `## 작업 분석` 섹션에 판단 결과 기록

## Phase 2: 코드베이스 탐색

**팀원 배정**: `team-explorer` 1~3명 (병렬)

**리더 액션**:
1. Task 도구로 team-explorer 에이전트를 **병렬** 실행 (각각 독립 임무)
2. 각 팀원에게 전달할 프롬프트 예시:
   ```
   [미션] (decisions.md에서 복사)
   [네 임무] 관련 기존 코드와 패턴을 분석하라. 핵심 파일 목록을 반환하라.
   ```
3. 팀원 결과 수신 → **핵심만 추출** → decisions.md에 기록
4. 팀원 **해고** (resume 하지 않음)

**decisions.md 업데이트**: 발견된 패턴, 핵심 파일 경로, 아키텍처 인사이트

---

## Phase 3: 설계 (필요시)

**팀원 배정**: `team-architect` 1명 / `team-db-expert` 1명 / `team-ui-expert` 1명 (작업 유형에 따라)

**리더 액션**:
1. decisions.md의 현재 내용을 요약하여 팀원에게 전달
2. Task 도구로 전문가 에이전트 실행:
   ```
   [미션 요약] (decisions.md에서)
   [탐색 결과 요약] (Phase 2 결과에서 핵심만)
   [네 임무] 위 정보를 바탕으로 구현 설계안을 수립하라.
   ```
3. 설계 결과 수신 → 핵심만 추출 → decisions.md에 기록
4. 팀원 **해고**

**decisions.md 업데이트**: `## 설계 결정` 섹션에 설계안, 파일 구조, 구현 순서 기록

---

## Phase 4: 구현

**팀원 배정**: `team-coder` (general-purpose 서브에이전트) — 작업 단위별로 새로 고용

**핵심 원칙**: 구현을 작은 단위로 쪼개서 각각 다른 팀원에게 배정한다.

**리더 액션**:
1. 설계안을 기반으로 구현 작업을 **작은 단위**로 분해
   - 예: "파일 A 생성", "파일 B의 메서드 X 수정", "파일 C에 이벤트 핸들러 추가"
2. 각 단위 작업마다 **새 팀원**을 Task 도구로 고용:
   ```
   [프로젝트 컨텍스트] ASP.NET Web Forms + DevExpress Bootstrap v25.1.6.0 + Oracle DB
   [프로젝트 규칙] CLAUDE.md 참조: 한글 주석, ASPX 주석은 <%-- --%>, MVC 패턴
   [설계안] (decisions.md의 설계 결정 섹션 요약)
   [네 임무] [구체적 작업 지시. 어떤 파일의 어떤 부분을 어떻게 변경할지 명확히]
   작업 완료 후 변경한 파일과 내용을 요약해서 보고하라.
   ```
3. 팀원 결과 수신 → decisions.md의 `## 완료된 작업`, `## 변경된 파일` 업데이트
4. 팀원 **해고**
5. 다음 단위 작업을 위해 **새 팀원 고용** (이전 팀원의 결과 요약만 전달)

**독립적인 구현 작업이 있으면 반드시 병렬로 실행한다.**

## Phase 5: 비판적 검토 (Critical Review)

**팀원 배정**: `team-critic` 1~2명 (병렬)

**CRITICAL: 이 단계는 절대 생략하지 않는다.**

**리더 액션**:
1. decisions.md에서 변경된 파일 목록을 확인
2. **새로운** team-critic 팀원을 Task 도구로 고용:
   ```
   [미션] (decisions.md에서)
   [변경된 파일 목록] (decisions.md에서)
   [네 임무] 위 파일들의 변경 내용을 비판적으로 검토하라.
   - 요구사항 충족 여부
   - 버그, 보안 취약점, 성능 이슈
   - CLAUDE.md 가이드라인 준수 여부 (한글 주석, ASPX 주석 형식 등)
   - 누락된 예외 처리
   CRITICAL/WARNING/SUGGESTION으로 분류하여 보고하라.
   ```
3. 검토 결과 수신 → decisions.md의 `## 발견된 이슈` 업데이트
4. 팀원 **해고**

---

## Phase 6: 개선 반영 (CRITICAL 이슈 있을 시)

**팀원 배정**: 새로운 `team-coder` (이슈 수정용)

**리더 액션**:
1. CRITICAL/WARNING 이슈가 있으면 **새 팀원** 고용하여 수정 지시:
   ```
   [프로젝트 컨텍스트] ...
   [수정할 이슈 목록] (decisions.md에서)
   [네 임무] 위 이슈들을 수정하라. 수정 내용을 보고하라.
   ```
2. 수정 결과 수신 → decisions.md 업데이트
3. 팀원 **해고**
4. CRITICAL 이슈가 있었다면 → Phase 5로 돌아가 **새 검토자** 고용하여 재검토
5. 모든 CRITICAL 해결 확인 시 다음 Phase로

**리더 액션**:

## 리더의 행동 규범

### 해야 하는 것
- decisions.md를 Read/Write/Edit 하는 것
- Task 도구로 팀원(서브에이전트)을 고용하는 것
- 팀원의 결과를 **요약**하여 decisions.md에 기록하는 것
- 사용자에게 진행 상황을 보고하는 것
- 작업을 잘게 쪼개서 각각 다른 팀원에게 배정하는 것

### 절대 하지 않는 것
- ❌ 소스 코드 파일을 직접 Read 하는 것
- ❌ 소스 코드를 직접 Edit/Write 하는 것
- ❌ Grep/Glob으로 코드를 직접 검색하는 것
- ❌ 끝난 팀원을 resume 하는 것
- ❌ 한 팀원에게 2개 이상의 큰 작업을 주는 것
- ❌ 팀원의 반환 결과를 통째로 기억하려는 것 (핵심만 추출)

### 팀원에게 주는 컨텍스트 템플릿
팀원을 새로 고용할 때 항상 다음 구조로 지시:
```
[프로젝트] KShift Smart Portal - ASP.NET Web Forms + DevExpress Bootstrap v25.1.6.0 + Oracle
[규칙] 한글 주석 필수, ASPX 주석: <%-- --%>, MVC 패턴, 복합키는 세미콜론 구분
[현재 상황] (decisions.md에서 관련 내용 요약)
[네 임무] (구체적이고 명확한 단일 작업)
[산출물] (기대하는 결과물 형태)
```

## 중요 규칙 요약

1. **리더는 전략만** — 코드는 한 줄도 직접 건드리지 않는다
2. **기억은 파일에** — 모든 결정과 진행을 decisions.md에 즉시 기록한다
3. **팀원은 일회용** — 작업 끝나면 해고, 새 작업은 새 팀원에게
4. **자율 판단** — 작업 유형을 사용자에게 묻지 않고 자동으로 판단한다
5. **병렬 실행** — 독립적인 작업은 항상 동시에 여러 팀원을 고용한다
6. **비판적 검토 필수** — 모든 구현은 반드시 critic 검토를 거친다
7. **반복 개선** — CRITICAL 이슈 발견 시 새 팀원으로 수정 후 새 검토자로 재검토
8. **한글 소통** — 사용자 보고와 decisions.md 모두 한글로
9. **최소 간섭** — 중간에 불필요한 질문 없이, 최종 결과만 보고한다
10. **컨벤션 준수** — CLAUDE.md 가이드라인을 팀원 지시에 항상 포함한다
